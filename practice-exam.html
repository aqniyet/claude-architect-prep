Scenario: Customer Support Resolution Agent
Question 1
Your agentic loop checks whether the assistant's response contains text to determine if the loop should terminate. In testing, the agent sometimes stops mid-task after producing a brief explanation before completing all necessary tool calls. What is the root cause?
AThe model's max_tokens is set too low, causing premature truncation.
BChecking for assistant text content is an unreliable completion indicator — the model may produce text between tool calls without intending to end the loop.
CThe system prompt doesn't explicitly instruct the model to complete all tool calls before responding.
DThe agentic loop needs a minimum iteration count before it's allowed to terminate.
Answer: B. The exam guide lists "checking for assistant text content as a completion indicator" as an anti-pattern. The correct approach is to check stop_reason: continue when "tool_use", terminate when "end_turn". Models may produce intermediate text alongside tool calls.
Question 2
Production data shows that in 8% of cases, your agent processes refunds without first verifying the customer's identity through get_customer. A senior engineer suggests strengthening the system prompt with "You MUST ALWAYS call get_customer before process_refund." What's the best approach?
AThe stronger system prompt wording is sufficient combined with few-shot examples showing the correct order.
BUse tool_choice forced selection to always call get_customer first on every turn.
CImplement a programmatic prerequisite that blocks process_refund until get_customer has returned a verified customer ID.
DDeploy a routing classifier that parses user messages and pre-selects the appropriate tool sequence.
Answer: C. When deterministic compliance is required for critical business logic (like identity verification before financial operations), prompt instructions have a non-zero failure rate. Programmatic enforcement via hooks or prerequisite gates provides deterministic guarantees.

Question 8

AThe error format doesn't use the MCP isError flag, so the agent doesn't recognize it as an error.
BUniform error responses prevent the agent from distinguishing retryable from non-retryable errors, blocking appropriate recovery decisions.
CThe message is too short for the model to understand the error context.
DError messages should include stack traces so the agent can debug the underlying issue.
Answer: B. The exam guide emphasizes returning structured error metadata including errorCategory (transient/validation/permission), isRetryable boolean, and human-readable descriptions. Generic errors hide whether the agent should retry, explain a policy violation, or escalate.
