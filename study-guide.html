Chapter 01

How Claude Code Is Configured

Claude Code is a tool that reads your codebase and follows instructions. But where do those instructions come from? This chapter explains the entire configuration system from first principles.

The Problem Configuration Solves

Imagine you have a team of 8 developers. You want Claude Code to:

  • Follow your team's coding standards (everyone)
  • Use different conventions for React components vs API handlers vs test files
  • Connect to your company's GitHub and Jira through MCP
  • Let one developer experiment with a personal database tool without affecting others
  • Run automatically in your CI pipeline with no human interaction

Each of these needs requires a different configuration mechanism. Claude Code doesn't have a single "settings file" — it has a layered system where different types of configuration live in different places, for good reason.

The Four Configuration Surfaces

Claude Code has four distinct things you can configure. Each one has its own file format and location:

SurfaceWhat It ContainsFormat
Instructions
(CLAUDE.md)		 Prose instructions: coding standards, conventions, architecture notes, what tools to prefer Markdown files
MCP Servers
(connection config)		 External tool servers: GitHub, Jira, databases, custom APIs JSON files
Commands
(slash commands)		 Reusable prompt templates invoked with /command-name Markdown files in directories
Skills
(advanced commands)		 Like commands but with isolation options, tool restrictions, and argument hints SKILL.md with YAML frontmatter

These are separate systems. Instructions go in CLAUDE.md, server connections go in JSON files, commands go in their own directory. They never mix. You would never put MCP server configuration inside CLAUDE.md, or put coding standards in a JSON config file.

Key Principle
Instructions = Markdown. Connections = JSON. Commands/Skills = their own directories. If an exam question mixes these up (e.g., "put server config in CLAUDE.md"), that option is wrong.

Shared vs Personal: The Two-Tier Rule

Every configuration surface has two tiers:

The Two-Tier Model
Shared
Lives in the project directory. Committed to git. Every teammate gets it when they clone.
↓
Personal
Lives in your home directory (~/). Never committed. Only you have it.

Here's the complete map:

SurfaceShared (Project)Personal (Home)
Instructions .claude/CLAUDE.md or root CLAUDE.md ~/.claude/CLAUDE.md
MCP Servers .mcp.json ~/.claude.json
Commands .claude/commands/ ~/.claude/commands/
Skills .claude/skills/ ~/.claude/skills/

The pattern is consistent: project-level config lives under .claude/ in your repo, personal config lives under ~/.claude/ in your home directory. The one exception is MCP: project = .mcp.json (at repo root), personal = ~/.claude.json.

Think About It
A new developer joins the team and Claude Code doesn't follow team standards. If the standards were in ~/.claude/CLAUDE.md, that's the problem — it's personal config, never shared via git. The fix is always: move it to project-level .claude/CLAUDE.md.

Both tiers are active at the same time. Claude Code loads your personal instructions AND the project instructions AND any directory-level instructions. They stack, they don't override each other. Similarly, tools from both personal and project MCP servers are all available simultaneously.

Path-Scoped Rules and When You Need Them

There's a fifth configuration location that deserves its own explanation: .claude/rules/.

Consider this situation: you want all test files to follow the same conventions. But test files are spread across your entire codebase — src/components/Button.test.tsx, src/api/auth.test.ts, lib/utils.test.js. You can't put a directory-level CLAUDE.md in every folder.

Path-scoped rules solve this. You create a file in .claude/rules/ with a YAML frontmatter block that specifies glob patterns:

# .claude/rules/testing.md --- paths: ["**/*.test.*", "**/*.spec.*"] --- Use describe/it blocks for all tests. Mock external services, never internal modules. Each test file needs at least one integration test.

These rules only load when Claude is editing a file that matches the glob pattern. If you're editing Button.tsx, the testing rules don't load. If you're editing Button.test.tsx, they do.

When to use which

SituationBest mechanism
Standards that apply to all code in the projectProject-level CLAUDE.md
Standards for one specific directory/packageDirectory-level CLAUDE.md in that directory
Standards for a file type spread across many directories.claude/rules/ with glob patterns

There's also @import syntax for keeping CLAUDE.md modular — you can reference external files from within CLAUDE.md to avoid one massive file.

Skills and Their Frontmatter

Skills are like advanced slash commands. They live in .claude/skills/ (shared) or ~/.claude/skills/ (personal) and have a SKILL.md file.

What makes skills different from regular commands is the frontmatter — YAML configuration at the top of SKILL.md that controls how the skill executes:

# .claude/skills/analyze-codebase/SKILL.md --- context: fork allowed-tools: ["Read", "Grep", "Glob"] argument-hint: "path to analyze" --- Analyze the given directory for dead code, circular dependencies, and unused exports. Return a structured report.

The three frontmatter options and what they do:

OptionWhat it doesWhen to use
context: fork Runs the skill in an isolated sub-agent. All the skill's output — verbose file listings, dependency graphs, exploration notes — stays in the fork. Only the final result comes back to your main conversation. Any skill that produces a lot of intermediate output you don't want cluttering your session.
allowed-tools Restricts which tools the skill can use. Acts as a safety boundary. When a skill should only read (not write), or shouldn't execute shell commands.
argument-hint Shows a prompt to the user when they invoke the skill without arguments. When the skill needs input (a file path, a feature name, etc).
Key Concept: context: fork
This is the answer whenever the exam describes verbose or exploratory output that shouldn't pollute the main conversation. Not "write to a file." Not "summarize before returning." The purpose-built mechanism is context: fork.

Running Claude Code in CI/CD

By default, Claude Code is interactive — it waits for user input in a terminal. In a CI/CD pipeline (GitHub Actions, Jenkins, etc.), there's no user. The job hangs.

The solution is one flag: -p (or --print).

# This hangs in CI (waits for interactive input): claude "Review this PR for security issues" # This works (processes and exits): claude -p "Review this PR for security issues" # For structured output that machines can parse: claude -p --output-format json "Review this PR" # For a specific JSON shape: claude -p --output-format json --json-schema schema.json "Review this PR"

There are only three CI-relevant flags. Everything else you might see on the exam (CLAUDE_HEADLESS=true, --batch, redirecting stdin) are fabricated distractors that don't exist.

FlagPurpose
-p / --printNon-interactive mode. Process, output, exit.
--output-format jsonOutput as JSON instead of plain text.
--json-schemaEnforce a specific JSON structure on the output.
Building On This
CI context also matters for review quality. A Claude session that generated code is less effective at reviewing its own changes — it retains its reasoning context and is less likely to question its own decisions. A separate, independent review instance is more effective. This connects to Domain 4's multi-pass review concepts.
Chapter 02

Getting Structured Data Out of Claude

Claude's natural output is text. But production systems need JSON, validated fields, and guaranteed schemas. This chapter explains how to bridge that gap reliably.

Why Text Output Breaks in Production

If you ask Claude to "return JSON," it usually will. But "usually" isn't good enough for a production pipeline that processes thousands of documents. Here's what goes wrong:

  • Syntax errors: Missing commas, unclosed brackets, trailing commas. Happens rarely, but at scale "rarely" means daily.
  • Format drift: The model wraps JSON in markdown code fences, or adds explanatory text before/after the JSON.
  • Schema violations: Missing required fields, wrong types, extra fields you didn't ask for.

You could try to fix these with workarounds: tell the prompt "return only JSON, no markdown," use regex to extract JSON blocks, or run a JSON repair library. These all treat symptoms. The exam consistently favors the approach that makes the problem structurally impossible.

tool_use as a Structured Output Mechanism

Here is the core insight: Claude's tool use feature was designed to call external functions, but it also serves as a structured output mechanism. When Claude calls a tool, it must produce JSON that exactly matches the tool's input schema. This is enforced at the system level — not by the prompt.

The pattern works like this:

Using tool_use for structured extraction
Step 1
Define a "tool" whose input schema IS your desired output format. The tool doesn't need to do anything — you just want the structured data.
↓
Step 2
Send your document/data to Claude along with the tool definition. Configure tool_choice to force the model to call the tool.
↓
Step 3
Read the structured data from the tool_use content block in the response. It's guaranteed to be valid JSON matching your schema.

Example: you want to extract invoice data. You define a tool called extract_invoice with an input schema containing fields like vendor_name, total_amount, line_items. When Claude "calls" this tool, it produces the extracted data as the tool's arguments — guaranteed to match your schema.

// Simplified example tools: [{ name: "extract_invoice", description: "Extract structured data from an invoice", input_schema: { type: "object", properties: { vendor_name: { type: "string" }, total_amount: { type: "number" }, line_items: { type: "array", items: { type: "object", properties: { description: { type: "string" }, amount: { type: "number" } } } } }, required: ["vendor_name", "total_amount", "line_items"] } }]
Key Distinction
tool_use eliminates syntax errors (malformed JSON). It does NOT prevent semantic errors (line items that don't sum to the total, values in wrong fields). For semantic validation, you still need a validation layer.

Controlling When the Model Calls Tools

The tool_choice parameter controls whether and which tools the model calls:

SettingBehaviorWhen to use
"auto" Model decides whether to call a tool or return text. Normal agent operation. No guarantee of structured output.
"any" Model must call a tool, but chooses which one. Multiple extraction schemas; document type is unknown. Guarantees structured output.
{"type":"tool", "name":"extract_invoice"} Model must call this specific tool. You know exactly what extraction to run. Maximum control.

For extraction pipelines where you always want structured output, use "any" or forced selection. "auto" lets the model return plain text instead, which defeats the purpose.

Designing Schemas That Prevent Hallucination

Schema design directly affects extraction quality. A poorly designed schema forces the model into hallucination.

The problem with all-required schemas: If your schema says fax_number is required, but the source document has no fax number, the model is trapped. It must produce a value to satisfy the schema, so it fabricates one. This isn't a prompt problem — it's a schema design problem.

The fix: Make fields nullable (optional) when the information might not exist in the source:

properties: { vendor_name: { type: "string" }, // Always on invoices total_amount: { type: "number" }, // Always on invoices fax_number: { type: ["string", "null"] // May not exist }, purchase_order: { type: ["string", "null"] // May not exist } } required: ["vendor_name", "total_amount"] // Only truly required fields

Now the model can return null for fax_number without penalty. No fabrication needed.

Other schema design patterns

SituationPattern
Categories where the source is ambiguous Add "unclear" to enum values
Categories that might not be in your predefined list Add "other" enum value + a detail string field
Cross-validation between fields Add both stated_total and calculated_total; compare downstream
Conflicting data in source Add a conflict_detected boolean field
Think About It
The exam tests this principle: prompt instructions ("don't fabricate") are weaker than schema design (nullable fields). The model can ignore a prompt instruction. It cannot ignore a schema that allows null — returning null becomes the easy, natural choice when information is absent.

When Extraction Fails: Validation and Retry

Even with tool_use, extractions can be semantically wrong (wrong values, misplaced data). The standard pattern is retry with error feedback:

Validation-Retry Loop
Extract
Send document + tool definition to Claude. Get structured output.
↓
Validate
Run semantic checks: do line items sum to total? Are dates valid? Are required fields non-null?
↓ (if validation fails)
Retry
Send the original document + the failed extraction + specific validation errors back to Claude.

The key detail: include the specific validation errors in the retry, not just "try again." This guides the model toward the exact problem.

But retries have limits. They work for:

  • Format mismatches (model put the date in the wrong field)
  • Structural errors (missed a nested object)
  • Arithmetic issues (sum doesn't match)

They do NOT work when the information simply isn't in the document. No amount of retrying will extract a fax number from a document that doesn't have one. Recognize the difference.

The Batch API and Its Constraints

The Message Batches API lets you send many requests at once for 50% less cost. But it has hard constraints that determine when you can and can't use it.

How it works

You submit a batch of requests, each with a custom_id. The API processes them and returns results, which you match back using the same custom_id. Processing can take up to 24 hours — there is no guaranteed latency.

The critical limitation

The Batch API does not support multi-turn tool calling within a single request. In a normal synchronous call, the model can call a tool, get the result, reason about it, call another tool, and so on. In a batch request, the model processes the input and returns output — no mid-request tool execution.

This means:

WorkflowCan use Batch?Why
Extract data from 500 invoices Yes Each invoice is a single input → single output. No tool calling needed.
Weekly technical debt report Depends If it's single-turn (all code provided in prompt), yes. If it needs to read files iteratively, no.
Pre-merge code review (blocking) No Developers wait for results. Up to 24h latency is unacceptable for blocking workflows.
Agent that reads files, then decides what to analyze next No Requires multi-turn tool calling. Must use synchronous API.
Decision Rule
Two questions determine if Batch API is appropriate:
1. Can the workflow tolerate up to 24 hours of latency? If no → synchronous.
2. Does the model need to act on tool results mid-request? If yes → synchronous.
Both must be "yes/no" for batch to work.
Chapter 03

Designing Tools and Distributing Them

Tools are how Claude interacts with external systems. How you design and distribute them directly affects whether the agent uses them correctly.

How Claude Chooses Which Tool to Call

When Claude has access to tools, it reads their names and descriptions to decide which one to use. This is the entire selection mechanism. There is no routing classifier, no keyword matching — just the model reading descriptions and making a judgment call.

This means tool descriptions are the single most important factor in tool selection reliability. A tool with a vague or minimal description will be misused. A tool with a clear, detailed description will be used correctly.

Writing Descriptions That Work

A minimal description:

// Bad: minimal, ambiguous { name: "get_customer", description: "Retrieves customer information" } { name: "lookup_order", description: "Retrieves order details" }

Both "retrieve information." If a user says "check my order #12345," the model might call either one. A good description includes:

  • What it does specifically
  • What inputs it accepts (formats, examples)
  • When to use it vs similar tools
  • What it returns
// Good: clear boundaries { name: "get_customer", description: "Look up a customer account by email address or phone number. Returns customer profile: name, account status, loyalty tier, contact info. Use this when you need to identify or verify a customer. Do NOT use for order-related queries." } { name: "lookup_order", description: "Look up order details by order ID (format: #12345) or tracking number. Returns: items, status, shipping info, payment. Use this when the customer asks about a specific order. Requires an order ID or tracking number." }

When tools have overlapping names (like analyze_content vs analyze_document), rename them to make the purpose obvious from the name alone: extract_web_results vs summarize_pdf.

The Tool Count Problem

There's a practical limit to how many tools an agent can handle effectively. The research and exam agree: 4-5 tools per agent is the sweet spot. At 18 tools, selection reliability degrades significantly.

Why? Each additional tool is another option the model must evaluate and distinguish. With 18 tools, especially if several have similar purposes, the decision space becomes too complex.

The solution: distribute tools across specialized agents

Instead of one agent with 18 tools, create multiple agents with focused tool sets:

Tool Distribution Pattern
Coordinator
Tools: Task (spawn subagents)
↓          ↓          ↓
Search Agent
3 tools
Analysis Agent
4 tools
Synthesis Agent
3 tools + verify_fact

Each agent gets only tools relevant to its role. This prevents cross-specialization misuse (e.g., a synthesis agent doing bad web searches when it should be synthesizing).

For high-frequency cross-role needs, you can give an agent a scoped tool. For example, the synthesis agent might get a verify_fact tool for quick lookups, while complex searches still route through the coordinator to the search agent.

Exam Principle
The exam favors reducing tool count over adding prompt instructions. If an agent misselects tools, the structural fix is fewer, better-described tools — not more few-shot examples on top of 18 confusing tools.

MCP Servers: Where Configuration Lives

MCP (Model Context Protocol) servers are external tool providers. They expose tools and resources that Claude can use. Configuration is covered in Chapter 1, but here are the details that matter for tool design:

  • Project-level (.mcp.json): Shared via version control. Use ${ENV_VAR} for secrets so tokens stay local.
  • User-level (~/.claude.json): Personal/experimental servers.
  • Tools from all configured servers are discovered at connection time and available simultaneously.

MCP resources are a separate concept from tools. Resources are content catalogs (issue summaries, doc hierarchies, database schemas) that give Claude visibility into available data without requiring exploratory tool calls. Think of resources as "what's available" and tools as "how to act."

When choosing whether to build a custom MCP server or use an existing one: prefer community/existing servers for standard integrations (GitHub, Jira). Build custom servers only for team-specific workflows.

Structured Errors for Intelligent Recovery

When a tool fails, the error response determines whether the agent can recover intelligently or is stuck guessing. A generic error like "Operation failed" tells the agent nothing.

Structured error responses include:

FieldPurposeExample
errorCategory Type of failure "transient", "validation", "permission", "business"
isRetryable Should the agent try again? true for timeouts, false for policy violations
description Human-readable explanation "Refund exceeds $500 policy limit"

This lets the agent make appropriate decisions: retry transient errors, explain business errors to the user, and escalate permission errors.

Important distinction: an access failure (timeout, service down) is different from a valid empty result (query ran successfully, found no matches). Returning empty results as success when the service actually failed prevents any recovery.

Chapter 04

Managing Context and Information

Claude processes everything in a context window — a fixed amount of text it can "see" at once. Managing what's in that window, and what happens when sessions get long, is essential for reliable behavior.

Why Context Degrades in Long Sessions

Two things happen as a conversation grows:

1. Summarization loses details. When the context window fills up, older content gets summarized. Summaries preserve themes but lose specifics — exact dollar amounts become "a refund," specific dates become "recently," and order numbers disappear entirely.

2. The "lost in the middle" effect. Models process information at the beginning and end of their context window more reliably than the middle. In a long document or conversation, findings from the middle may be overlooked.

3. Tool results accumulate. Each tool call returns data that stays in context. A get_customer call might return 40+ fields when only 5 are relevant. After many calls, irrelevant data crowds out important information.

Session Management: Resume, Fresh Start, Fork

Claude Code offers three ways to handle sessions across work periods:

Resume (--resume)

Continues a prior session. The full conversation history is available. Use this when the prior context is mostly still valid — you haven't made major changes to the code being discussed.

If specific files changed, you can inform the resumed session about just those changes rather than re-exploring everything.

Fresh session with injected summary

Start a brand new conversation, but include a structured summary of what you learned before. The model gets zero stale context — no outdated function signatures, no references to deleted files. Use this when significant code changes happened since the last session.

Fork (fork_session)

Creates independent branches from a shared analysis baseline. Both branches have the same history up to the fork point, then diverge. Use this to explore two different approaches from the same starting analysis (e.g., comparing testing strategies or refactoring approaches).

Decision Rule

Prior context mostly valid → --resume

Code changed significantly → new session + summary

Want to compare approaches → fork_session

Context window filling during active work → /compact

/compact compresses the current session to free up space. It's for managing size during active work, not for fixing stale data. Stale data stays stale even after compaction.

Preserving Facts Across Summarization

The core problem: a customer support session involves three issues with specific dollar amounts, order numbers, and dates. As the conversation grows and gets summarized, "$47.99 on order #8823" becomes "the billing issue" and the agent starts confusing it with the $23.50 credit on a different order.

The solution is to separate facts from conversation. Extract critical structured data into a persistent "case facts" block that lives outside the summarized history:

// Case facts block (persisted in every prompt) CASE FACTS: - Issue 1: Missing item on order #8823, credit requested: $23.50 - Issue 2: Billing discrepancy on order #9102, overcharge: $47.99 - Issue 3: Address update to 142 Oak Street - Customer: Jane Smith (ID: C-44821), loyalty: Gold tier

This block is included in every prompt, never summarized. The conversation history can be compressed freely because the critical facts are preserved separately.

Related techniques:

  • Trim tool outputs before they accumulate: keep only relevant fields from verbose API responses.
  • Position key information at the start of long inputs to mitigate the "lost in the middle" effect.
  • Scratchpad files for long exploration sessions: agents write findings to a file and reference it later, countering context degradation.

Provenance: Keeping Track of Where Things Came From

In a multi-agent research system, information flows from source → search agent → analysis agent → synthesis agent → final report. At each step, attribution can be lost. "AI adoption is at 35%" means nothing without knowing who said it, when, and based on what.

Claim-source mappings

The foundational technique is requiring subagents to output structured claim-source pairs:

{ claim: "AI adoption in enterprises reached 35%", source: "McKinsey Global Survey 2024", url: "https://...", publication_date: "2024-03-15", methodology: "Survey of 1,200 enterprises globally" }

Downstream agents must preserve these mappings through synthesis, not compress them into unsourced statements.

Handling conflicts

When credible sources disagree (Source A says 35%, Source B says 52%), the synthesis agent's job is to faithfully represent the disagreement, not resolve it:

  • Include both values with source attribution
  • Annotate why they might differ (different definitions, different time periods, different methodologies)
  • Structure the final report to distinguish well-established findings from contested ones

The synthesis agent does NOT escalate conflicts to the coordinator (that's for errors and failures) and does NOT pick one source over another. Conflicting data is normal research output, not an error condition.

Temporal data

Requiring publication dates in structured outputs prevents temporal differences from being misinterpreted as contradictions. "35% in 2022" and "52% in 2024" aren't conflicting — they show growth.

Connecting The Dots
Provenance ties back to schema design from Chapter 2. The synthesis agent's output schema should have fields for source, date, and confidence — and these should be required fields (since sources should always be cited). Meanwhile, uncertain facts should use the patterns from Chapter 2: "unclear" enums, conflict_detected booleans, separate stated_value and calculated_value fields.
Summary

The Exam's Decision Framework

Across all domains, the exam applies a consistent philosophy. Understanding it helps even on questions you haven't studied.

The Solution Hierarchy

When a question offers multiple solutions that could work, the exam ranks them by reliability:

Preferred → Last Resort
Structural
Make the problem impossible by design.
tool_use schemas, nullable fields, tool reduction, fresh sessions
↓
Programmatic
Enforce rules with code guarantees.
Hooks, prerequisite gates, validation loops
↓
Prompt-Based
Guide behavior probabilistically.
Few-shot examples, system prompt instructions, explicit criteria

Prompt-based solutions aren't wrong — they're the right answer when no structural or programmatic option exists. But when a structural option IS available, it always wins.

Practice applying this: for each exam question, categorize every option as structural, programmatic, or prompt-based. If there's a structural option, it's almost certainly correct.
