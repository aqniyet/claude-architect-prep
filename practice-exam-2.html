Agentic Architecture & Orchestration

Scenario: Multi-Agent Research System
Question 1
Your research coordinator delegates to web search, document analysis, and synthesis subagents sequentially — waiting for each to finish before spawning the next. A typical research task takes 180 seconds end-to-end. The subagents don't have dependencies between them; they just need the same query. What's the most effective optimization?
AIncrease the max_tokens on each subagent so they produce more comprehensive output in fewer iterations.
BCombine the three subagents into a single agent with all tools available so only one Task call is needed.
CHave the coordinator emit multiple Task tool calls in a single response to spawn the three subagents in parallel.
DCache subagent results so repeated queries skip the full execution path.
Answer: C. Parallel subagent execution is achieved by emitting multiple Task tool calls in a single coordinator response, not across separate turns. The three subagents run concurrently, dramatically reducing latency for independent work. Option B violates specialization and exceeds tool count limits.
Question 2
A stakeholder complains that your research reports on broad topics (e.g., "impact of climate policy") consistently miss major subtopics. Examining logs, you see the coordinator decomposes tasks with prompts like "find climate papers" and "summarize climate news." The subagents execute correctly, but coverage is incomplete. Where should you focus your fix?
AThe synthesis agent — add instructions for it to identify and flag coverage gaps.
BThe coordinator — its task decomposition is too narrow, creating subagent assignments that don't cover all relevant dimensions of the topic.
CThe web search agent — its queries are too generic and need domain-specific expansion.
DThe document analysis agent — it's filtering out relevant sources during analysis.
Answer: B. When subagents execute their assigned work correctly but overall coverage is incomplete, the root cause is the coordinator's task decomposition. If the coordinator never asks "climate policy + agriculture" or "climate policy + labor markets," no downstream agent can recover those gaps. Iterative refinement loops in the coordinator (evaluating synthesis output and re-delegating to fill gaps) address this.
Scenario: Customer Support Resolution Agent
Question 3
Your support agent integrates with three backend systems through MCP tools. Each system returns timestamps in a different format: Unix epoch seconds from billing, ISO 8601 from orders, and date strings like "March 15, 2026" from shipping. The agent frequently confuses dates and makes incorrect comparisons. What's the best approach?
AImplement a PostToolUse hook that normalizes all timestamps to a single format before the model processes them.
BAdd system prompt instructions explaining each format with conversion examples so the model can interpret them correctly.
CAdd a convert_timestamp tool that the model can call whenever it needs to compare dates across systems.
DRequest that each backend team standardize their APIs to a common timestamp format.
Answer: A. PostToolUse hooks intercept tool results for transformation before the model processes them — exactly the pattern for normalizing heterogeneous data formats from different MCP tools. This provides deterministic normalization without adding agent overhead or requiring the model to remember conversion rules.
Question 4
Your support agent successfully resolves a complex billing issue, but the customer needs an escalation for a separate policy exception. When handing off to a human agent who cannot see the conversation transcript, what should the agent provide?
AA full transcript of the conversation, so the human agent has complete context.
BThe customer's email address so the human agent can re-initiate contact.
CA structured handoff summary with customer ID, root cause analysis, refund amount (if any), and a recommended action.
DOnly the policy exception request, so the human can investigate without bias from prior context.
Answer: C. Structured handoff summaries (customer ID, root cause, amounts, recommended action) give human agents the critical facts without forcing them to parse a full transcript. Transcripts require reading time and still miss the "what was concluded" synthesis. Structured handoffs are the documented pattern for mid-process escalation.

Question 6
Your code review agent processes 14-file pull requests in a single pass and produces inconsistent results: thorough analysis for some files, superficial comments for others, and occasionally contradictory feedback between files. How should you restructure the review workflow?
AForce developers to submit smaller PRs of at most 3-4 files to ensure consistent analysis.
BSwitch to a model with a larger context window to give all files adequate attention.

Question 7
Your process_refund MCP tool has a business rule: refunds over $500 must go to a supervisor, not be processed automatically. Currently, you rely on the system prompt to instruct the agent not to auto-refund large amounts. Occasionally the agent still processes a $700 refund. What's the most reliable fix?
AAdd few-shot examples showing the agent escalating all refunds over $500 rather than processing them.
BImplement a tool call interception hook that blocks process_refund calls above $500 and redirects to an escalation workflow.
CRephrase the system prompt with stronger language: "You MUST NEVER process refunds over $500 without supervisor approval."
DRemove the process_refund tool from the agent entirely and route all refunds through humans.
Answer: B. Tool call interception hooks provide deterministic enforcement for policy-violating actions. Prompt-based options (A and C) are probabilistic — they reduce frequency but don't guarantee compliance, which is unacceptable when money is involved. Option D overcorrects by removing a useful capability.

Scenario: Structured Data Extraction
Question 9
You have a single analyze_document MCP tool used across extraction, summarization, and verification workflows. The agent frequently uses it for the wrong purpose — e.g., summarizing when extraction was needed, or extracting when summary was requested. What's the best fix?
AAdd a required mode parameter to analyze_document with values "extract", "summarize", or "verify".
BSplit the generic tool into purpose-specific tools: extract_data_points, summarize_content, and verify_claim_against_source.
CImprove the system prompt with detailed instructions on when to use each mode of analyze_document.
DAdd few-shot examples showing correct use of analyze_document for each purpose.
Answer: B. Splitting generic tools into purpose-specific tools with defined input/output contracts is the documented pattern. A tool named extract_data_points is self-documenting — the agent selects it when it needs extraction. A mode parameter (A) still requires the agent to guess correctly. Prompt-based fixes (C, D) don't address the root ambiguity in the tool interface itself.
Question 10
Your extraction pipeline needs to reliably call exactly one of three extraction tools based on document type. The model occasionally returns conversational text instead of calling any extraction tool. How should you configure tool_choice?
AUse tool_choice: "auto" and add prompt instructions requiring a tool call.
BUse tool_choice: {"type": "tool", "name": "extract_invoice"} as a default that the model can override.
CUse tool_choice: "required" (a new option) to force any tool call.
DUse tool_choice: "any" to guarantee the model calls a tool while letting it choose which one.

"auto" (A) permits conversational text. Option B forces a specific tool even when it's wrong. Option C is fabricated — "required" is not a valid tool_choice value.

After reviewing 120 pages of source material, your analysis agent correctly identifies findings from the first 30 pages and last 20 pages but overlooks material from the middle sections. Which mitigation strategy best addresses this?
AAUpgrade to a model with a larger context window — more tokens will improve middle-section attention.
BReduce the source material to only the most relevant 50 pages before analysis.
CPlace key findings summaries at the beginning of aggregated inputs and organize detailed results with explicit section headers.
DIncrease the max_tokens so the model has room to mention all sections.
Answer: C. The "lost in the middle" effect means models process information at the beginning and end more reliably than the middle. Option D addresses output length, not input attention.
Question 20
Your document analysis subagent encounters a source with conflicting statistics: the abstract says "30% increase" and Figure 3 in the same paper shows a 45% increase. How should the subagent report this?
AEscalate the document to the coordinator for manual reconciliation before continuing.
BComplete the analysis with both values included, explicitly annotated as conflicting, letting the coordinator decide how to handle before passing to synthesis.
CUse the value from the abstract since abstracts are typically the most authoritative source.
DFlag the document as unreliable and exclude it from the final report.
Answer: B. The documented pattern: complete document analysis with conflicting values included and explicitly annotated, letting the coordinator decide how to reconcile before passing to synthesis. This preserves information provenance (both values + their locations) and enables downstream decisions. Silently picking one (C) loses data; excluding the document (D) loses valid findings.
